Our Commitment

Security and privacy are core values of Paperlyte. We take the security of our code, our infrastructure, and our users' data seriously. This page summarizes our security practices and explains how to report a vulnerability. For the complete, technical policy, see our full Security Policy on GitHub.

1. Supported Versions

We release patches for security vulnerabilities for the following versions:

Version Supported
0.x.x ✅ Yes

As we are in early development (pre-1.0), we currently support only the latest version. Once we reach 1.0, we will provide security updates for multiple versions.

2. Reporting a Vulnerability

If you believe you have found a security vulnerability in Paperlyte, we encourage you to let us know right away through responsible disclosure.

Report security issues to
security@paperlyte.com

Please do

  • Email us privately with a detailed description, steps to reproduce, and the potential impact of the issue.
  • Include the type of vulnerability, affected files or locations, and any proof-of-concept if available.
  • Give us reasonable time to investigate and address the issue.

Please do not

  • Open a public GitHub issue for security vulnerabilities.
  • Disclose the vulnerability publicly before it has been addressed.
  • Access, modify, or destroy data that is not yours, or exploit the issue in any way.

You may also use GitHub's private vulnerability reporting feature as an alternative to email.

3. Responsible Disclosure Process

When you report an issue, here is what you can expect from us:

  • Acknowledgment of your report within 48 hours.
  • A detailed response within 7 days as we investigate and validate the issue.
  • Resolution and coordinated disclosure, including crediting you in our security advisory if you wish.

4. Current Security Measures

  • TypeScript strict mode to catch type-related bugs at compile time.
  • Static analysis with ESLint — TypeScript ESLint, React Hooks, and React Refresh rules to catch common mistakes early.
  • Content Security Policy (CSP) to help protect against cross-site scripting (XSS).
  • Secure external links using rel="noopener noreferrer".
  • Dependency hygiene — a pinned dependency lockfile for deterministic, reproducible builds.

5. Planned Improvements

  • End-to-end encryption for note data
  • Two-factor authentication (2FA)
  • Regular security audits and penetration testing

6. Current Limitations

As we are in early development, some protections are not yet in place, including user authentication, encryption at rest, and backend API security. These will be addressed as the project matures. For the full and most current details, please refer to our Security Policy on GitHub.

7. Questions

If you have questions about our security practices or a report you have submitted, contact us at security@paperlyte.com.
